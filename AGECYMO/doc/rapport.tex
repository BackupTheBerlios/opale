	\section{Introduction}

	Le principe de ce projet est de mettre en place une application capable de déterminer si un modèle 
	donné est valide ou non. La validité d'un modèle, dans ce projet, est définie selon trois critères :
	\begin{itemize}
	\item[-] le nombre d'intersections d'arètes en un point à l'intérieur des arêtes
	\item[-] le nombre de faces ayant des sommets non-coplanaires
	\item[-] le nombre d'intersections d'arètes et de faces en un point intérieur.
	\end{itemize}
	Plus ces valeurs sont proches de zéro, plus le modèle sera dit \emph{valide}. 
	Afin de calculer ces différentes valeurs, l'application propose deux méthodes :
	\begin{itemize}
	\item[-] la première, regroupée dans un plugin du nom de \textbf{mantyla}, permet d'effectuer des calculs 
		en acceptant les erreurs selon une tolérance spécifiée par l'utilisateur.
	\item[-] la seconde, regroupée au sein du plugin nommé \textbf{core},
		qui utilise la librairie core qui permet d'effectuer des calculs en
		précision exacte.
	\end{itemize}

	Dans le but de parfaire notre projet de \emph{construction de cylindre
		généralisé}, nous avons donc choisi d'ajouter le test de validité
	sous la forme de deux plugins indépendants (l'un pour \textbf{core} l'autre
		pour \textbf{mantyla}).  Ainsi, les modèles chargés,
	ainsi que les modèles construits via le moteur du cylindre généralisé
	peuvent être soumis aux différents tests de validité avec les deux
		plugins au sein de la	même application.


	\section{Modèles testés, résultats et interprétations}

	Nous avons effectué les tests de validation sur les modèles suivants :
	\begin{itemize}
	\item Un cube (cf figure~\ref{cube}) avec deux faces non  coplanaires
	\item Un vase (cf figure~\ref{vase})
	\item Le lapin simplifié de Standford (cf figure~\ref{bunny})
	\item Le dragon simplifié de Standford (cf figure~\ref{dragon})
	\item Une main simplifiée de Standford (cf figure~\ref{hand})
	\item Une fourmi (cf figure~\ref{fourmille})
	\item Des pièces d'échecs (cf figures~\ref{chess1},\ref{chess2},
	\ref{chess3} et~\ref{chess4}).
	\end{itemize}

	testé, le dénombrement des cas dégénérés pour chacun des 3 tests et
	ce pour \textbf{mantyla} et pour \textbf{core}. Le nombre de cas non-dégénérés est
	donc égal au nombre de vérifications moins le nombre de cas dégénérés.

	\section{Discussion}

	Comme cela a été précisé en introduction, la validation des modèles se déroule en trois étapes distinctes :

	\begin{itemize}
	\item[-] Dans un premier temps, un test est effectué pour déterminer
		le nombre d'intersections non valides existantes entre les arêtes du
		modèle considéré. Entre deux arêtes, il existe 3 cas possibles :
		
		\begin{itemize}
		\item[-] CAS 1 : les deux arêtes ne s'intersectent en aucune
		façon, le test d'intersection va alors considérer la situation valide
		
	\item[-] CAS 2 : les deux arêtes s'intersectent au niveau de leurs
	extrémités, le test d'intersection va aussi considérer cette
	situation comme valide
	
	\item[-] CAS 3 : les deux arêtes s'intersectent en un point
	situé à l'intérieur d'au moins une des deux arêtes, le test
	d'intersection va alors repérer cette situation comme non valide.
		\end{itemize}

		Cependant, si les calculs sont effectués avec le plugin
		\textbf{mantyla}, l'utilisation d'une tolérance modifie légèrement
		l'appréciation de ces 3 cas. En effet, la tolérance est une valeur
		qui va permettre au test de validité d'être plus \emph{perméable}
		aux  éventuelles erreurs des modèles. Ainsi, si la tolérance est
		fixée à $0.2$, le point de coordonnées $(0.0;0.5;1.0)$ sera
		confondu avec le point de coordonnées $(0.2;0.3;1.1)$. De ce fait,
		en ce qui concerne l'intersection entre deux arêtes, si un modèle
		contient des arêtes qui ne s'intersectent pas véritablement en
		leur extrémités, avec une tolérance suffisamment élevée,
		l'intersection se révélera au final valide. Il est donc logique
		de croire, qu'une tolérance très faible (10-10) ne vas révéler que les
		intersections \emph{réelles} c'est à dire que les intersections
		existantes physiquement dans le modèle . Plus on augmente la
		tolérance et plus on va trouver de nouvelles intersections. Le
		principe serait alors de trouver la bonne valeur pour la
		tolérance, afin de révéler d'une part les intersections
		\emph{réelles} ainsi que les intersections \emph{théoriques} c'est
		à dire celles qui devraient exister. Attention toutefois à ne pas
		augmenter de trop la tolérance, pour ne pas créer des
		intersections indésirables. 

	\item[-] Dans un second temps, un test est effectué sur chacune des
	faces afin de définir si les sommets associés sont tous
	coplanaires. Il est évident qu'une face triangulaire est
	obligatoirement coplanaire. Le problème n'est donc présent que pour
	des faces ayant au moins quatre sommets. Avec le plugin
	\textbf{mantyla}, la notion de tolérance est aussi présente. Plus
	cette dernière est élevée, plus une face a de chances d'être coplanaire.

	\item[-] Enfin, le dernier test consiste à trouver le nombre
	d'intersections entre  les arêtes et les faces en un point
	intérieur à une face. Afin de simplifier l'algorithme, nous avons considéré 
	uniquement des faces convexes. Les mêmes
	remarques concernant \textbf{mantyla} et la tolérance sont applicables ici. 

	\end{itemize}


	\section{Interprétation et conclusion}
	
	\begin{itemize}

	\item[-] Test 1 : intersections non valides entre arêtes

	Le fait d'augmenter le epsilon sur les modèles testés a
	pour but de baisser le nombre d'erreurs. Plus l'epsilon
	est élevé, plus la taille des deux segments testés diminue
	à partir de ses extrémités. Pour une valeur supérieure ou
	égale à $0.5$, les deux extrémités du segment fusionnent. Il
	est donc normale pour les résultats récupérés de n'avoir 
	aucune erreur supérieure à $0.5$.\\
	Néanmoins une variation anormale se produit sur certains 
	modèles pour des valeurs de epsilon faibles (Main, fourmi, pièces
	d'échecs). \\
	En effet, ceci peut s'expliquer via le fait que l'epsilon
	peut aussi engendrer des erreurs. En effet, plus l'epsilon est 
	augmenté et plus il y aura des intersections entre les arêtes. 
	Cependant, il y a des intersections considérées comme valides
	(si l'intersection s'effectue aux extrémités du segment) et
	des intersections dîtes non valides (si l'intersection ne 
	s'effectue pas au niveau des extrémités). Le nombre d'intersection
	non valide peut donc complètement varier d'un epsilon à un autre et
	ne suit en aucun cas une logique bien définie. Par exemple un epsilon à 
	$0.2$ peut considérer que deux arêtes (A et B) ne se touche pas, un epsilon
	à $0.3$ peut considérer que A et B s'intersectent de façon non valide, et un 
	epsilon de $0.4$ lui considérera que l'intersection entre A et B est finalement
	valide.\\
	


	\item[-] Test 2 : Coplanarité des faces

	  Plus le epsilon est élevé, plus les tests admettent une coplanarité
	  inexacte. Ce principe s'applique sur les valeurs entre $0$ et $0.5$. Il 
	  est détecté sur les tests une remontée significative
	  des erreurs pour un epsilon entre $0.5$ et $1.0$. \\
	  Si l'epsilon est supérieur à $0.5$, tous les points de la
	  face sont considérés comme étant identiques (de part le fait
	  qu'un segment s'étend entre $0.0$ et $1.0$). Il est donc
	  logique de trouver un nombre d'erreur 
	  conséquent lorsque l'epsilon dépasse la valeur $0.5$.\\
	  

	\item[-] Test 3 : intersection arêtes-faces
	  
	  Les erreurs varient ici de deux façons selon les modèles :
	  \begin{itemize}
	  \item[-] plus l'epsilon est élevé, plus les erreurs décroissent
	    (Main, Vase, Lapin, Dragon).
	  \item[-] les erreurs admettent un nombre significatif d'erreurs
	    pour les epsilons $0.0$, $0.5$ et $1.0$ (Pièces d'échec, fourmis).
	  \end{itemize}
	  De nouveau nous sommes confrontés aux problèmes liés à la notion
	  de l'epsilon.\\
	  	  
	\item[-]Test Core :

	  Les résultats du test \textbf{core} donne pour tous les tests un nombre
	  d'erreurs moyen par rapport aux tests \textbf{mantyla}. Il semblerai que
	  ce nombre d'erreurs moyen corresponde à un epsilon optimum.
	  
	\end{itemize}
